# Extend the Hypostructure Framework

Read `references/extension-work-packet.md` and complete it from live evidence.
Implement the smallest reusable capability that makes the real consumer thin.

## Prove the gap

Read the consumer mathematics, API contracts, migration row, exact current
source, fixtures, and import firewall. Demonstrate that no live public
declaration already supplies the capability. Distinguish an absent feature from
an inconvenient interface and from an application-specific lemma.

Apply a hard evidence gate: a declaration counts as a live framework capability
only when it is owned by the `Hypostructure.*` namespace in the `hypostructure/`
source tree (or an explicitly approved new-package source). A declaration or
consumer under `StructuralExhaustion.*`, the old Lean framework subtree, or a
legacy application tree is parity evidence only, even if it already executes a
similarly named route. Do not use a legacy canonical registry to infer the live
Hypostructure owner or status. Resolve the live namespace and source path before
declaring a gap closed.

Do not copy legacy source. Legacy behavior may define parity observations, but
the new implementation must satisfy Hypostructure's ownership, residual,
execution, and trust contracts independently.

## Select the owner

- Put domain-neutral finite, residual, coordinate, assembly, budget, normal-form,
  response, metadata, and closure machinery in Core.
- Put one tactic's mathematics and machine in that `CTN` namespace.
- Put typed CT-to-CT transitions in Routes.
- Put parameterized finite graph semantics in Graph.
- Put represented analytic semantics in PDE.
- Keep fixed constants and theorem-specific arithmetic in the application.

Create a decision record from `migration/hypostructure/decisions/0000-template.md`
for a public API, ownership, compatibility, or trust-boundary decision.

## Minimize the author surface

Accept only primitive definitions, operators, exact residual-owned enumerations,
deciders, semantic laws, local certificates, and explicitly approved imported
contracts. Derive branch choices, search results, residuals, traces, work counts,
provenance, and closure inside the framework. Record the split with
`Core.Provision` and complete `Core.Metadata`.

Never let a capability accept its expected result, terminal equality, trace,
route, successor, final theorem, or a detached replacement carrier.

## Complete the vertical slice

For a CT, implement in order: `Spec`, `Capability`, `State`, `Search`,
`Execution`, `Theorems`, and `Automation`; then add domain constructors and
fixtures. Prove all outcomes, soundness, totality, determinism where promised,
and the actual polynomial work envelope.

For a route, register a typed `Core.Routing.Profile` under an existing or newly
reviewed `Routes.Registry.Entry`. A catalog row alone is not execution. Preserve
the literal source stage, perform semantic discovery, execute the real target
capability, extend the ledger once, and expose provenance.

For Core, Graph, or PDE, expose only the minimum interface needed by at least
one real consumer. Avoid compatibility facades, reverse imports, application
names, and declarations meaningful only during migration.

## Require transfer and every residual

Add an axiom-free neutral fixture whenever possible. Add Graph and PDE fixtures
when the abstraction is meaningful in those domains. A graph-only or PDE-only
capability needs a second named application or finite model proving reuse. Pin
deterministic order, every complementary residual, exact traces, and check counts.

## Update evidence honestly

Run focused builds, package builds, the import firewall, trust checks, and
`#print axioms`. Refresh migration records only after source and fresh kernel
evidence exist; never promote mathematical, parity, work, or integration status
from file presence alone. Finish with `$review-hypostructure-framework-change`.
