# Repair a Hypostructure Manuscript

Repair the failed implication as an explicit residual branch. Preserve the theorem and existing topology until a separate repair sketch passes adversarial review. Require an exact incoming stage, a typed consumer for every outcome, and an explicit termination or routing-rank argument for every cycle.

## EG authority preflight

For any Erdős--Gyárfás Problem 64 task, read
`original_erdos_64_proof.tex` **FIRST**, before any API/process document,
migration row, generated view, living proof, or legacy Lean source. It is the
immutable sole authority for EG mathematics, strategy, node identity and
responsibility, and DAG topology. Freeze the exact quantified node contract,
branch alternatives, and exact incoming/outgoing DAG edges from that file.

Only after that contract and those edges are frozen may the matching
kernel-checked legacy `NodeX.lean` be read, and then only as implementation and
parity evidence. `proofs/erdos_64_eg/erdos_64_proof.tex` is a living,
non-binding editorial cross-check; it cannot repair, supplement, weaken,
strengthen, or redirect the original contract or edges. Any discrepancy blocks
the task: record it and stop rather than reconciling the sources or silently
changing the obligation. Never edit `original_erdos_64_proof.tex`.

After this preflight, API/process documents govern framework ownership,
capabilities, migration process, and status only. They never outrank or
reinterpret the original on EG mathematics, node responsibility, or DAG
topology.

## Establish current capability

Before drafting, read:

- the exact theorem, failed node, all incoming/outgoing paths, and dependent conclusions;
- `DOMAIN_INDEPENDENT_CORE.md`, the applicable Graph/PDE API, and the repair/ownership sections of `HYPOSTRUCTURE_MIGRATION_GUIDE.md`;
- [`references/repair-worksheet.md`](references/repair-worksheet.md);
- the relevant API/node/PDE matrix rows; and
- the live Core, CT, Routes, Graph, or PDE declarations and fixtures implicated by the proposed repair.

Use the methodology in `framework/branch_closure_methodology_extended.tex` for invariant selection and branch closure, but translate every implementation obligation to current Hypostructure APIs. Treat legacy Lean only as parity/counterexample evidence. Never import it into production or assume a legacy handoff exists in Hypostructure.

Specifications describe the target architecture; matrices, source, focused elaboration, and `.olean` freshness determine current availability. Never design a merged repair around a `planned` executor.

## Freeze the defect

Record the exact quantified implication, smallest verified ancestor, literal predecessor residual, complete accumulated ledger queries, all outgoing consumers, blast radius, and residual created by negating the conclusion. Do not weaken the theorem, add the missing result as a hypothesis, or borrow a fact from a sibling/later branch.

Keep the proof ledger distinct from any mathematical `ClosedClassLedger`: the first preserves proved facts about one residual; the second records classes removed from later quotient-compatible invariants. Do not use one as a substitute for the other.

## Normalize and attack the claim

Write all domains, codomains, dependencies, and quantifiers literally. Test especially:

- injection versus realization/surjection;
- pairwise separation versus one simultaneous witness;
- representative existence versus quotient well-definedness;
- coordinate independence versus a complete product/cube; and
- a local context generator versus every compatible global context.

Negate the exact claim and extract a canonical residual-owned witness: first failure, least missing class, rank drop, overloaded fibre, minimal obstruction, zero-gap profile, or target-visible quotient. Prove its selection is deterministic or provide a locally checkable certificate.

## Select a repair invariant

Inventory only resources available in the incoming ledger and attach a precise price/account to each. For every candidate predicate record:

1. what its positive side proves or pays;
2. what bounded/represented residual its negative side emits;
3. how it is computed from the literal residual; and
4. which existing CT/domain executor consumes each side.

Apply the Graph plus PDE/neutral both-sides test. Reject a predicate that simply renames the desired conclusion. Prefer the earliest exact CT contract and keep branches separate when payloads or consumers differ.

## Route only through available Hypostructure

For every proposed step, inspect the CT vertical slice and executable route in live source. Supply only irreducible `Core.Provision` inputs. Require the framework to construct decisions, target inputs, traces, routes, ledger extensions, coordinates, assembly, and closure.

Every leaf must have either:

- direct `Core.Closure` evidence;
- strict-progress closure;
- a resource contradiction;
- an exact allowlisted imported contract;
- an acyclic reduction with strict routing-rank descent; or
- a typed active residual with a named current consumer.

If the mathematics is sound but no public executor expresses it, use [`../extend-hypostructure-framework/SKILL.md`](../extend-hypostructure-framework/SKILL.md). Create an ADR, identify the owner, author primitives, generated outputs, every residual, Graph and PDE/neutral fixtures, work, metadata, and trust. Do not place temporary route, ledger, coordinate, quotient, gluing, or handoff plumbing in the application.

## Derive missing local lemmas

Treat each absent local implication as a repair task rather than a caller-supplied premise:

1. restate it using only available ledger inputs;
2. negate it and seek the least finite/represented counter-witness;
3. run the both-sides test on that witness;
4. try the lowest correct existing Core/CT/domain theorem;
5. record any false subimplication and countermodel;
6. refine the residual, invariant granularity, or exact consumer; and
7. extend the framework only for a reusable expressibility gap.

Continue until all leaves close or a precise active residual remains. Do not report “the user must prove the lemma” as a completed repair.

## Draft, prototype, and red-team

Draft under `proofs/<problem>/repairs/` while leaving the source manuscript unchanged. Include the frozen state, canonical witness, both-sides table, CT/executor table, complete leaf table, persistence, local computation/work, trust boundary, and proposed TeX/Lean statements.

Prototype reusable logic at its Core/CT/Routes/Graph/PDE owner with positive and complementary fixtures. Keep the application to fixed data and a thin executor call. Never enumerate an ambient graph family or continuum, replace the incoming carrier, or raise resource limits.

Invoke [`$red-team-hypostructure-manuscript-repair`](../red-team-hypostructure-manuscript-repair/SKILL.md) on the actual sketch. A FAIL blocks manuscript edits. Repair the sketch and rerun until PASS.

## Merge after PASS

Only after PASS:

1. integrate the proved repair without weakening the theorem or silently changing topology;
2. update the diagram/dependency, invariant, residual, and defect records owned by mutable repair or editorial documents; for Erdős--Gyárfás Problem 64 this never authorizes editing `original_erdos_64_proof.tex`;
3. synchronize Lean executors and metadata;
4. update API/node/PDE matrices without conflating kernel, parity, math, work, integration, and publication status;
5. add normalized parity evidence where migration behavior changes; and
6. run focused builds, the import/trust audit, domain fixtures, manuscript checks, and applicable migration integration checks.

Report the failed implication, exact residual, selected invariant, both sides, CT/domain route, all closure mechanisms/active consumers, framework changes, red-team verdict, trust delta, work bound, synchronization, and commands.
